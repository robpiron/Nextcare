  1. ARCHITETTURA ENTERPRISE — MULTI-TENANT & MULTI-SEDE
================================================================================

MODELLO A 3 LIVELLI
-------------------
Il sistema deve implementare una topologia gerarchica a 3 livelli:

  LIVELLO 0 — GRUPPO (Console centrale)
  ├── Dashboard consolidata inter-sede
  ├── Configurazione globale (nomenclatore, listini master, medici)
  ├── Reportistica aggregata su tutte le sedi
  ├── Fatturazione centralizzata e prima nota di gruppo
  └── Gestione utenti globale + propagazione regole ai livelli inferiori

  LIVELLO 1 — SEDI OPERATIVE
  ├── Ogni sede ha: agenda, medici, analizzatori, PACS, CUP, contabilità locale
  ├── Dati clinici isolati per sede (GDPR)
  ├── KPI e reportistica fluiscono verso il Livello 0
  └── Configurazioni locali sovrascrivono selettivamente quelle globali

  LIVELLO 2 — PUNTI PRELIEVO SATELLITE
  ├── Stazioni leggere browser-based (tablet + stampante barcode)
  ├── Accettazione campioni, stampa etichette, CUP locale
  ├── Campioni tracciati fino all'analizzatore del laboratorio principale
  └── Attivazione nuovo punto: < 15 minuti, zero IT on-site


MULTI-TENANT
------------
  - Ogni tenant (cliente) ha DATABASE FISICAMENTE SEPARATO (schema isolation o
    DB separato in modalità high-security)
  - Nessuna query cross-tenant possibile a livello architetturale
  - Il provider gestisce un template base; ogni tenant sovrascrive localmente
  - Aggiornamenti: deploy rolling, zero downtime, tutti i tenant aggiornati
    simultaneamente
  - Nomenclatore 2025: distribuzione automatica a tutti i tenant alla data di
    entrata in vigore
  - Console reseller: visibilità su tutti i tenant, sconti volume progressivi

MULTI-SEDE (dentro lo stesso tenant)
-------------------------------------
  - Medici itineranti: agenda unificata con anti-double-booking inter-sede
  - Listini differenziati per sede (privato / SSN / fondi / aziendale)
  - Report consolidato con KPI comparativi drill-down per sede
  - LIS inter-sede: campione prelevato in Sede B → analizzato in Sede A
    con tracciabilità completa del percorso

PUNTI PRELIEVO SATELLITE — REQUISITI TECNICI
----------------------------------------------
  - Interfaccia: browser-based, responsive touch (tablet iOS/Android/Windows)
  - Stampa etichette: Zebra ZPL/EPL, Brother, Dymo (TCP raw porta 9100)
  - Barcode campione: code128 (o GS1-128), attributi univoci:
      paziente + data + esame + sede prelievo + operatore
  - 7 stati campione:
      prelievo → trasporto → ricevimento lab → analisi
      → refertazione → firma → consegna
  - Aggiornamento stato via scansione barcode
  - Alert automatici per campione non conforme (emolizzato, insufficiente)
  - CUP integrato: prenotazione diretta su agenda sede principale

  3. STANDARD SANITARI — HL7 / FHIR / ASTM / DICOM

------------------------------------------------------------------------
  3.1 HL7 v2.x (versioni 2.3 – 2.8)
------------------------------------------------------------------------
  Protocollo trasporto: MLLP (Minimal Lower Layer Protocol) su TCP/IP
  Framing MLLP:
    Start Block:   0x0B (Vertical Tab)
    End Block:     0x1C (File Separator) + 0x0D (Carriage Return)
    Formato:       <0x0B>MSH|...<0x1C><0x0D>

  Delimitatori messaggio:
    |   separatore campi
    ^   separatore componenti
    ~   separatore ripetizioni
    \   carattere escape
    &   separatore sub-componenti

  Messaggi utilizzati da NextCare:
    ADT (Cap.3)  — Ammissioni, Dimissioni, Trasferimenti, Anagrafiche
    ORM (Cap.4)  — Ordini (laboratorio, radiologia)
    OML          — Ordini laboratorio (versione moderna)
    ORU (Cap.7)  — Risultati osservazione (refertazione)
    MDM          — Documenti clinici

  Versione raccomandata: 2.5.1 (CLIA compliant) — verificare spec del vendor
  Z-segment: per estensioni custom (non standard)
  Sicurezza: VPN o TLS sul socket MLLP (HL7 non ha auth nativa)
  URL docs:  https://www.hl7.org
             https://caristix.com/hl7 (riferimento segmenti, utile in sviluppo)


------------------------------------------------------------------------
  3.2 HL7 FHIR R4 (v4.0.1)
------------------------------------------------------------------------
  Protocollo:    REST/HTTPS
  Formato:       JSON (preferito) o XML o Turtle
  Autenticazione: OAuth 2.0 (SMART on FHIR)
  Endpoint metadata: [base]/metadata → restituisce CapabilityStatement

  Operazioni REST standard:
    read      GET    [base]/[ResourceType]/[id]
    search    GET    [base]/[ResourceType]?param=val
    create    POST   [base]/[ResourceType]
    update    PUT    [base]/[ResourceType]/[id]
    delete    DELETE [base]/[ResourceType]/[id]
    transaction POST [base] (Bundle)

  Resource principali usate da NextCare:
    Patient              — Anagrafica paziente
    Observation          — Referti LIS (risultati esame)
    DiagnosticReport     — Referto firmato
    Encounter            — Accettazione / visita
    ServiceRequest       — Ordine prestazione (sostituisce ORM)
    Practitioner         — Medico
    Organization         — Sede / struttura
    ImagingStudy         — Studio DICOM collegato
    Appointment          — Prenotazione CUP
    Coverage             — Fondo sanitario / assicurazione

  Search parameters utili:
    _id, _lastUpdated, _count, _include, _revinclude

  FHIR Operations:
    $validate, $everything (Patient summary), $match (MDM)

  Utilizzo in NextCare:
    - Portale MMG: accesso referti via FHIR R4
    - FSE 2.0: trasmissione documenti (in evoluzione da CDA2 a FHIR)
    - Azure Health Data Services: FHIR server cloud

  URL docs: https://www.hl7.org/fhir/R4/index.html
            https://www.hl7.org/fhir/R4/http.html


------------------------------------------------------------------------
  3.3 DICOM 3.0 — MODALITY WORKLIST (MWL)
------------------------------------------------------------------------
  Protocollo:    DICOM su TCP/IP
  Operazione:    C-FIND (query worklist)
  SOP Class MWL: UID 1.2.840.10008.5.1.4.31

  Ruoli:
    SCU (Service Class User)  → modalità (TC, MRI, US) che interroga
    SCP (Service Class Provider) → RIS / Worklist Manager che risponde

  Flusso tipico:
    1. La modalità apre Association DICOM → negozia SOP Class MWL
    2. Invia C-FIND-RQ con attributi di ricerca (data, modalità, AE Title)
    3. Il SCP risponde con C-FIND-RSP per ogni worklist item trovato
    4. La modalità acquisisce e invia immagini → C-STORE verso PACS

  Attributi chiave query MWL:
    Scheduled Station AE Title
    Scheduled Procedure Step Start Date (range date)
    Modality (CT, MR, US, CR, DX, OT...)
    Accession Number
    Patient ID / Patient Name

  SOP Class Storage SCU/SCP: per ricevere e archiviare immagini
  Query/Retrieve: C-MOVE / C-GET per recupero immagini da PACS
  DICOM SR: Structured Report (per referti radiologici strutturati)
  TLS: opzionale ma raccomandato (DICOM TLS su porta diversa dalla standard)

  Tool di test/sviluppo:
    DCMTK (wlmscpfs, storescu, storescp) — https://dicom.offis.de/dcmtk.php.en
    Orthanc (plugin Worklists) — https://www.orthanc-server.com

  Conformance Statement: ogni apparecchio pubblica il proprio — richiederlo al vendor
  URL standard: https://www.dicomstandard.org (NEMA)
                PS3.4 Annex K (MWL SOP Class)
                PS3.3 IODs (Information Object Definitions)


------------------------------------------------------------------------
  3.4 ASTM E1394 / CLSI LIS2-A2
------------------------------------------------------------------------
  Nota: ASTM E1394 è formalmente ritirato (ASTM ~2013).
        Il successore ufficiale è CLSI LIS2-A2, ma la terminologia
        "ASTM" rimane prevalente negli analizzatori clinici.

  Protocollo trasporto:
    Legacy:  RS-232 seriale
    Moderno: TCP/IP socket (wrapper dei messaggi ASTM)

  Formato messaggi (record-based ASCII):
    H  — Header (inizio sessione, mittente, destinatario)
    P  — Patient (dati paziente)
    O  — Order (ordine esame)
    R  — Result (risultato)
    C  — Comment (commento)
    L  — Terminator (fine sessione)
    Delimitatori: | \ ^ &

  Flusso bidirezionale:
    HOST → ANALIZZATORE: Download ordini (O record per ogni test)
    ANALIZZATORE → HOST: Upload risultati (R record per ogni parametro)

  Note:
    Ogni analizzatore pubblica la propria Interface Specification.
    RICHIEDERE SEMPRE la spec dell'interfaccia al vendor dello strumento.
  URL: https://www.clsi.org (CLSI LIS2-A2 — a pagamento)


------------------------------------------------------------------------
  3.5 FSE 2.0 — FASCICOLO SANITARIO ELETTRONICO
------------------------------------------------------------------------
  Formato documenti:   HL7 CDA2 (Clinical Document Architecture v2)
                       In evoluzione verso HL7 FHIR R4
  Protocollo:          HTTPS/SOAP (Gateway FSE 2.0)
                       IHE XDS.b per condivisione documenti
  Autenticazione:      SAML v2.0 (asserzioni di sicurezza INI)
                       Certificati digitali per accreditamento

  Profili IHE utilizzati:
    XDS.b: ITI-41 (Provide & Register), ITI-18 (Registry Stored Query)
    PIX, PDQ (in contesto Affinity Domain Italia)

  Servizi Gateway NextCare deve supportare:
    - Validazione documento CDA2
    - Pubblicazione (invio al FSE regionale)
    - Sostituzione / aggiornamento metadati
    - Eliminazione documento
    - Ricerca e recupero

  OBBLIGATORIO: accreditamento presso AgID prima del go-live in produzione.

  Specifiche INI: versione 2.6.x (aggiornamenti frequenti — monitorare)
  URL: https://www.fascicolosanitario.gov.it (area tecnica)
       https://developers.italia.it
       https://www.agid.gov.it

================================================================================
  7. FONDI SANITARI & ASSICURAZIONI
================================================================================

FLUSSO DI INTEGRAZIONE
-----------------------
  1. Accettazione paziente → NextCare identifica la copertura (fondo/assicurazione)
  2. Verifica eleggibilità in tempo reale (API del fondo → risposta < 2 sec)
  3. Preautorizzazione automatica (per prestazioni previste dal piano)
  4. Erogazione prestazione
  5. Invio automatico documentazione di rimborso al fondo
  6. Riconciliazione pagamenti e aggiornamento scadenziari

FONDI INTEGRATI
---------------
  UniSalute (Gruppo Unipol)      — rimborsi diretti, preauth online
  Previmedical                   — voucher, preautorizzazione, rete diretta

  9. RBAC — RUOLI E PERMESSI GRANULARI
================================================================================

ARCHITETTURA
------------
  6 profili predefiniti + ruoli custom con permessi a livello di singola funzione
  Audit log immutabile (GDPR art. 30): chi ha fatto cosa, quando, da quale IP

RUOLI PREDEFINITI
-----------------

  AMMINISTRATORE
    Accesso: completo (configurazioni, utenti, reportistica finanziaria)
    Figure: titolare, direttore sanitario
    Operazioni permesse: tutte

  MEDICO
    Accesso: refertazione, agenda propria, cartelle cliniche propri pazienti
    Non vede: dati finanziari, cassa, contabilità
    Limitazione: visibilità agenda inter-sede configurabile

  LABORATORISTA (BIOLOGO / TECNICO)
    Accesso: LIS completo (accettazione campioni, analisi, validazione, refertazione)
    Non vede: agenda visite, fatturazione

  ACCETTAZIONE (SEGRETARIA / FRONT-DESK)
    Accesso: CUP, prenotazioni, accettazione, cassa
    Non vede: referti clinici completi, dati laboratorio

  CONTABILITÀ
    Accesso: fatturazione, pagamenti, corrispettivi, prima nota
    Non vede: dati clinici

  MAGAZZINO
    Accesso: gestione reagenti, consumabili, ordini fornitori, cold chain
    Non vede: dati clinici, fatturazione

  RUOLO CUSTOM
    Combinazione libera di permessi per modulo/funzione
    Configurabile dall'Amministratore

AUDIT LOG
---------
  Campi registrati per ogni operazione:
    - Utente (ID + nome)
    - Timestamp (UTC)
    - IP sorgente
    - Azione eseguita
    - Entità modificata (ID + tipo)
    - Valore prima della modifica
    - Valore dopo la modifica

  Il log è in SOLA LETTURA anche per l'Amministratore.
  Non può essere alterato né eliminato.
  Retention: configurabile (default 10 anni)
  Export: CSV per ispezioni Garante Privacy

 13. SPID — IDENTITÀ DIGITALE

PROTOCOLLI
----------
  SAML 2.0:      profilo Web Browser SSO (nativo SPID)
  OpenID Connect: (OIDC — Linee Guida AgID n.616/2021, alternativa moderna)

IMPLEMENTAZIONE SAML 2.0
-------------------------
  1. App Registration come Service Provider (SP) presso AgID
  2. Generare coppia chiavi RSA-2048 + certificato X.509 (RFC-5280)
  3. Produrre Metadata SP (XML firmato)
  4. Registrare SP nel Registry SPID: https://registry.spid.gov.it
  5. Configurare tutti gli Identity Provider (IDP) federati
     (Aruba, Infocert, Poste, TIM, ecc.)
  6. Implementare:
     - AuthnRequest (firmata XML → inviata a IdP)
     - Response/Assertion (firmata dall'IdP → validata da NextCare)
  7. Testare con Metadata Validator: https://validator.spid.gov.it
  8. Tool open source: https://github.com/italia/spid-saml-check

LIVELLI DI SICUREZZA
---------------------
  SpidL1 — password semplice
  SpidL2 — OTP / app autenticazione (raccomandato per dati sanitari)
  SpidL3 — smart card / firma digitale qualificata

 14. STAMPA BARCODE & ETICHETTE (ZPL)
================================================================================

LINGUAGGIO ZPL II (Zebra Programming Language)
-----------------------------------------------
  Struttura etichetta base:
    ^XA                          (Start label)
    ^FO[x],[y]^FD[testo]^FS     (Field Origin + Field Data + Field Separator)
    ^BY[largh_barre],[moltipl],[altezza]   (Barcode parameters)
    ^BC[orient],[altezza],[HRI],[linea_sup],[check]  (Code 128)
    ^BQ[orient],[model],[errore]  (QR Code)
    ^XZ                          (End label)

  Comandi principali:
    ^FO  — Field Origin (posizione x,y in dot)
    ^FD  — Field Data (contenuto campo)
    ^FS  — Field Separator (fine campo)
    ^BY  — Barcode parameters
    ^BC  — Code 128 barcode
    ^BQ  — QR Code
    ^GF  — Graphic Field (immagine raster)
    ^CF  — Change Font (font, dimensione)
    ^PQ  — Print Quantity (quante etichette stampare)

  Formato barcode campione (campi obbligatori):
    ID paziente (CF o ID interno)
    Data prelievo (YYYYMMDD)
    Codice esame (codice nomenclatore)
    ID sede prelievo
    Numero progressivo campione
    Operatore (ID utente)

STAMPA NETWORK
--------------
  Protocollo: TCP raw porta 9100 (invio diretto stringa ZPL)
  LPR/LPD:   porta 515 (alternativa legacy)
  USB:        driver installato su client
  Compatibilità:
    Zebra (ZT, ZD, GK series) — ZPL nativo
    Brother (QL, PT series)   — supporta ZPL o ESC/P
    Dymo (LabelWriter)        — driver ZPL o applicazione API

SIMULATORE PER SVILUPPO
-----------------------
  Labelary (web): https://labelary.com/viewer.html
  Anteprima ZPL senza stampante fisica.

URL docs: https://www.zebra.com (ZPL II Programming Guide — PDF)
          Titolo: "ZPL II, ZBI 2, Set-Get-Do, Mirror, WML Programming Guide"
